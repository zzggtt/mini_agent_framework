# 阶段一到阶段四学习版 Walkthrough

## 1. 这份文档的目的

这不是一份功能清单，也不是简单的开发日志，而是一份面向学习的源码导读。

我会重点解释三件事：

1. 每个阶段想解决什么问题。
2. 为什么当前阶段要先做这个，而不是直接做后面的能力。
3. 当前代码里的核心设计，后面会怎样继续长出来。

如果你在学习 Agent 框架，这四个阶段最重要的不是“写了多少代码”，而是逐步建立下面这条主线：

```text
项目骨架
-> 统一消息结构
-> 稳定的测试替身
-> 最小可用的 Agent 主循环
```

也可以换一种更偏运行链路的角度来理解：

```text
Message
-> FakeLLM
-> Agent State
-> Context Assembly
-> Assistant Reply
```

目前阶段一到阶段四，代码已经隐含出 4 层结构：

## 3. 阶段一：项目骨架为什么先做

阶段一看起来代码最少，但它其实解决的是最基础的工程问题：

```text
后面写的代码要放在哪里？
怎么运行？
怎么测试？
怎么让 IDE 和 pytest 正确识别？
```

### 3.1 当前已经落下的骨架

项目现在的最小结构大致是：

```text
my_agent_framework/
├── my_agents/
│   ├── core/
│   └── agents/
├── tests/
├── examples/
├── prd.md
└── pyproject.toml
```

### 3.2 为什么 `pyproject.toml` 很重要

`pyproject.toml` 当前内容很简单，但它做了三件事：

1. 告诉工具这是一个 Python 项目。
2. 声明 Python 版本和依赖。
3. 让 `uv run pytest`、`uv run python ...` 这些命令有统一的运行入口。

你现在的依赖包括：

- `pytest`
- `openai`
- `python-dotenv`
- `pydantic`

注意，这一阶段虽然还没真正接入 `openai`，但先把依赖放进来有两个好处：

1. 后面接真实 `LLM` 不用再重做环境准备。
2. 项目方向会更清晰，不会写到一半才补依赖。

### 3.3 为什么还要有 `tests/conftest.py`

这其实是阶段一和阶段二之间非常关键的工程胶水。

文件：

- `tests/conftest.py`

这个文件的作用是把项目根目录加入 `sys.path`，让测试里能正常导入：

```python
from my_agents.core.message import Message
```

如果没有它，pytest 很容易报：

```text
ModuleNotFoundError: No module named 'my_agents'
```

所以阶段一虽然不是业务逻辑，但它保证了后面每个阶段都能被验证。

### 3.4 阶段一真正解决的问题

阶段一真正解决的不是“建几个目录”，而是：

- 给后续代码一个稳定归属地
- 给测试一个可运行环境
- 给示例脚本一个可演示入口

换句话说，阶段一把“学习项目”变成了“能持续生长的工程项目”。

## 4. 阶段二：为什么先做 Message

如果你直接写 Agent，很容易一上来就在代码里到处拼这样的 dict：

```python
{"role": "user", "content": "你好"}
```

短期看能跑，长期看会有几个问题：

1. role 到处是字符串，容易拼错。
2. 历史消息、工具消息、系统消息没有统一类型。
3. 后续调试、测试、转换 API 格式时都很散。

所以阶段二先做 `Message`，本质上是在解决：

```text
消息应该长什么样，谁来统一管理它？
```

### 4.1 `MessageRole` 的设计目的

文件：

- `my_agents/core/message.py`

`MessageRole` 用 `Enum`，而不是裸字符串，核心价值是“约束”。

它当前支持：

- `system`
- `user`
- `assistant`
- `tool`

为什么现在就把 `tool` 放进去？

因为虽然阶段四还没开始工具调用，但整个消息系统要从一开始就按完整对话协议来设计。

这属于一种很好的“最小前瞻性设计”：

- 不做复杂工具调用逻辑
- 但先把消息角色边界定好

### 4.2 `Message` 为什么是 dataclass

`Message` 目前有两个字段：

- `role`
- `content`

用 `@dataclass(slots=True)` 的好处是：

1. 写法简洁，适合这种纯数据对象。
2. 语义清楚，读代码时一眼就知道它是“消息数据”。
3. `slots=True` 让对象更紧凑，也避免随手挂额外属性。

这里的重点不是性能，而是：

```text
Message 是一种明确、稳定、受约束的数据结构
```

### 4.3 `to_dict()` 在设计上为什么关键

你现在内部统一用 `Message` 对象，但真实聊天模型一般吃的是：

```python
{"role": "user", "content": "你好"}
```

所以 `to_dict()` 的作用是把“内部表示”转换为“外部协议”。

这一层非常重要，因为它把：

- 框架内部设计
- 模型 API 输入协议

分开了。

以后即使你要换 Provider，内部 Agent 逻辑也不一定需要跟着大改。

### 4.4 `from_dict()` 为什么不是多余的

很多人第一次写这种结构时只写 `to_dict()`，不写 `from_dict()`。

但 `from_dict()` 在学习阶段有两个好处：

1. 测试时可以验证“对象 <-> 字典”的双向转换。
2. 后续如果你要做历史恢复、日志回放、消息落盘，会更方便。

它不是当前最核心的能力，但属于很低成本、很有价值的补充。

### 4.5 阶段二真正解决的问题

阶段二不是在定义一个普通数据类，而是在定义：

```text
Agent 框架内部统一的上下文单元
```

后面这些东西最终都会变成 `Message`：

- 用户输入
- 模型回复
- system prompt
- 工具结果
- 记忆注入
- RAG 检索片段

这就是为什么 `Message` 要先于 `Agent` 实现。

## 5. 阶段三：为什么必须先有 FakeLLM

很多新人学 Agent 时，最容易踩的坑是过早接真实模型。

一旦直接接真实模型，你的测试就会受到这些因素影响：

- API Key
- 网络
- 费用
- 模型随机性
- 服务不稳定

这会导致你根本分不清：

```text
是我 Agent 写错了，还是模型今天输出变了？
```

所以阶段三先做 `FakeLLM`，它解决的是：

```text
如何在不依赖真实模型的前提下，稳定测试 Agent 主循环
```

### 5.1 FakeLLM 的最小接口为什么只有 `chat(messages)`

文件：

- `tests/fakes.py`

`FakeLLM` 当前只保留一个最小接口：

```python
chat(messages) -> str
```

这是非常刻意的设计。

因为阶段三的目标不是模拟完整 OpenAI SDK，而是只模拟当前阶段真正需要的接口契约。

这样做的好处是：

1. 接口极小，学习成本低。
2. 测试目标明确，不会被无关细节干扰。
3. 后续真实 `LLM` 只要也实现这个接口，就能无缝替换。

### 5.2 为什么 `responses` 要做成队列

`FakeLLM(["第一条", "第二条"])` 的意义是：

- 第一次 `chat()` 返回第一条
- 第二次 `chat()` 返回第二条

这样你就能测试多轮对话。

如果只返回固定常量，虽然也能测首次调用，但没法很好地验证“第二轮是不是带着历史继续聊”。

所以回复队列是为了支持：

```text
多轮可预测输出
```

### 5.3 为什么要记录 `calls`

这是 `FakeLLM` 最有价值的地方之一。

`calls` 会记录每次传给模型的 `messages`。

这意味着你后面测 `SimpleAgent` 时，不只是能断言“返回值对不对”，还可以断言：

- system prompt 有没有被拼进去
- 第一轮 user 消息有没有传进去
- 第一轮 assistant 消息有没有进入第二轮上下文

这就把测试从“黑盒”变成了“半白盒”。

换句话说，`FakeLLM` 不是只为了给出假回复，而是为了让上下文编排变得可观察。

### 5.4 为什么队列耗尽要抛错

当前实现里，如果回复队列为空，再调用 `chat()` 会触发 `IndexError`。

这其实是一个很好的默认行为。

因为它会让测试快速失败，而不是静默返回 `None`。

静默失败往往最难排查。

### 5.5 阶段三真正解决的问题

阶段三真正做成的是：

```text
把“测试 Agent”这件事从“依赖模型服务”改成了“依赖可控契约”
```

这一层做好了，阶段四就能专注于 Agent 逻辑本身。

## 6. 阶段四：最小 Agent 主循环是怎么成立的

阶段四是目前最关键的一步。

因为前面三步只是准备：

- 项目能运行
- 消息有统一结构
- 模型调用可被替身模拟

直到阶段四，才第一次真正做出：

```text
一个可以对话的 Agent
```

### 6.1 为什么要拆成 `Agent` 和 `SimpleAgent`

很多新人第一次写 Agent 会直接写一个类，把历史、模型、run 流程全堆进去。

这样短期能跑，但后面扩展会很痛苦。

因为未来你还会有：

- `ToolAgent`
- `ReActAgent`
- 可能还有记忆注入、RAG 增强等变体

这些 Agent 都会共享一些基础状态：

- `name`
- `llm`
- `system_prompt`
- `_history`

所以阶段四的关键设计就是把：

- 公共状态
- 公共历史操作

放到基类 `Agent` 中。

而把“这一轮怎么跑”的逻辑留给 `SimpleAgent`。

### 6.2 `Agent` 基类到底承担什么角色

文件：

- `my_agents/core/agent.py`

当前 `Agent` 基类做了 4 件事：

1. 保存公共依赖和配置
2. 保存 `_history`
3. 提供历史读写接口
4. 规定子类必须实现 `run()`

这个设计的核心思想是：

```text
公共状态统一管理，具体流程延后分化
```

这样以后你写 `ToolAgent` 时，不用重新发明一套历史管理。

### 6.3 为什么 `_history` 不包含 `system_prompt`

这是阶段四里很值得学的一个点。

当前设计里：

- `_history` 只保存真实会话消息
- `system_prompt` 是独立配置

然后每次 `run()` 时动态把 `system_prompt` 拼到最前面。

这样设计的好处：

1. `system_prompt` 不会在历史里重复堆积。
2. `get_history()` 返回的是纯会话历史，更直观。
3. 以后如果要换 prompt 组装策略，改动更局部。

所以这里其实是在区分三个不同概念：

- `system_prompt`：运行配置
- `_history`：会话状态
- `messages`：本轮实际发给模型的上下文

这三个东西一定不要混。

### 6.4 `get_history()` 为什么返回副本

这也是非常经典的封装设计：

```python
return list(self._history)
```

而不是：

```python
return self._history
```

原因是保护内部状态。

如果你直接把 `_history` 暴露出去，外部代码可能会无意中把它清掉、改掉、打乱顺序。

这会让 Agent 的状态变得不可控。

所以这里虽然只是一个小实现，但背后是很典型的“封装边界”设计。

### 6.5 `SimpleAgent.run()` 为什么这样写

文件：

- `my_agents/agents/simple_agent.py`

`run()` 现在的主流程非常简洁，但这正是阶段四最核心的价值。

#### 第一步：把用户输入包装成 `Message`

这一步的意义是让框架内部统一使用消息对象，而不是到处传字符串。

#### 第二步：先把 user 消息写进历史

这样本轮调用模型时，就能直接基于完整历史来构造上下文。

这是一个很重要的设计选择：

- 不是单独临时拼当前输入
- 而是先纳入历史，再统一组装消息

这会让后续更复杂的 Agent 也保持同样的模式。

#### 第三步：组装本轮消息

现在的上下文构造是：

```text
system prompt + history
```

也就是说，模型看到的永远不只是当前一句话，而是一个被组织过的上下文序列。

#### 第四步：调用 `llm.chat(messages)`

这一层故意只依赖抽象接口，不依赖具体 OpenAI SDK。

所以当前可以接：

- `FakeLLM`

后面也能接：

- 真实 `LLM`

这说明阶段四的设计已经具备了很好的“接口替换性”。

#### 第五步：把 assistant 回复写回历史

这是多轮对话成立的关键。

如果不保存 assistant 回复，下一轮只有 user 输入，模型就看不到之前自己说过什么。

所以连续对话依赖的是：

```text
user 和 assistant 往返都进入 history
```

### 6.6 阶段四真正解决的问题

阶段四真正做成的不是“能输出一句话”，而是：

```text
让 Agent 第一次拥有完整的一轮执行闭环
```

这个闭环就是：

```text
输入
-> 写入历史
-> 组装上下文
-> 调模型
-> 保存回复
-> 返回结果
```

后面的 ToolAgent、ReActAgent 只是在这个闭环基础上继续扩展。

## 7. 测试在这四个阶段里扮演什么角色

这四个阶段里，测试不是“最后补一下”，而是和设计一起推进的。

### 7.1 阶段二测试在验证什么

`tests/test_message.py` 不是只在测语法，它在保护 4 个设计点：

1. 角色受约束
2. 消息可构造
3. 能转成 API 协议格式
4. 非法角色不会悄悄混进系统

### 7.2 阶段三测试在验证什么

`tests/test_llm.py` 保护的是 `FakeLLM` 的契约：

1. 回复顺序可预测
2. 调用记录可观察
3. 队列耗尽快速失败

这意味着你后面测试 Agent 时，才能信任这个替身。

### 7.3 阶段四测试在验证什么

`tests/test_agent.py` 保护的是整个主循环：

1. 首轮对话能返回
2. 第二轮上下文里确实带上了历史
3. `_history` 的结构正确
4. 状态可以被清空

其中最有价值的一条是“检查第二轮传给 `FakeLLM` 的 messages”。

这条测试不是只看结果，而是在验证：

```text
上下文编排逻辑是否正确
```

这恰恰是 Agent 框架里最核心的部分。

## 8. 示例脚本为什么也值得看

文件：

- `examples/01_simple_chat.py`

这个示例不是为了炫功能，而是为了把阶段四主循环可视化。

它演示了：

1. 第一轮回复正常返回
2. 第二轮能基于上一轮历史继续回复
3. `get_history()` 的长度会随着对话累积增长

你运行它时看到最后输出是 `4`，其实就说明当前历史里有：

1. 第一轮 user
2. 第一轮 assistant
3. 第二轮 user
4. 第二轮 assistant

这正好把 `_history` 的设计落成了一个很直观的结果。

## 9. 把四个阶段串起来看

如果你把阶段一到阶段四串起来，会看到一种很清晰的递进关系。

### 阶段一：先把地基打好

解决：

- 项目能不能跑
- 测试能不能跑
- 代码往哪放

### 阶段二：统一消息表示

解决：

- 上下文怎么表达
- 消息如何与模型 API 对接

### 阶段三：提供稳定模型替身

解决：

- 没有真实 LLM 时怎么测试

### 阶段四：实现最小 Agent 主循环

解决：

- 谁来组织消息
- 谁来维护历史
- 一轮对话如何闭环

它们之间的承接关系可以写成：

```text
阶段一：准备项目骨架
阶段二：定义消息结构
阶段三：定义模型替身
阶段四：把消息和模型替身组织成一个可运行的 Agent
```

## 10. 当前阶段最值得记住的核心思路

如果只记 5 句话，我建议记这 5 句：

1. `Message` 不是普通数据类，而是整个上下文系统的最小单元。
2. `FakeLLM` 的价值不在于“像模型”，而在于“可控、可观察、可测试”。
3. `Agent` 基类负责公共状态，`SimpleAgent` 负责最小执行流程。
4. `system_prompt`、`history`、`messages` 是三个不同层次的概念。
5. Agent 框架的本质不是“调一次模型”，而是“组织上下文并维护循环状态”。

## 12. 一句话回顾

阶段一到阶段四最重要的成果，不是“写了几个文件”，而是你已经把 Agent 框架最核心的骨架拆出来了：

```text
先有项目骨架，
再统一消息结构，
再引入稳定测试替身，
最后做出最小 Agent 主循环。
```

只要这条主线真正理解了，后面的工具调用、ReAct、Memory、RAG 都是在这个骨架上继续生长。
